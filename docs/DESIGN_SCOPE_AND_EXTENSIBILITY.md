# 物资范围隔离与系统扩展设计

## 一、目标

1. **电气 / 机械前端隔离**：电气人员只看电气相关菜单与页面，机械人员只看机械相关；无“同时维护电气和机械”的场景。
2. **资源占用可控**：不因多套物资类型而让单用户加载全部代码，避免项目臃肿、内存与 CPU 占用过高。
3. **部署后可持续扩展**：后续新增物资类型（如生活消耗品）时，尽量支持在**部署后**通过配置或小范围发布完成，而非每次大改、全量发版。

---

## 二、推荐方案：按用户物资范围隔离 + 路由懒加载

### 2.1 用户物资范围（material_scopes）

- **含义**：每个用户拥有一个“物资范围”列表，表示可访问的物资类型。
- **取值**：`electrical`（电气）、`mechanical`（机械），后续可扩展 `consumables`（生活消耗品）等。
- **存储**：用户表增加字段 `material_scopes`，例如：
  - 类型：`VARCHAR(200)` 或 `JSON`（如 `["electrical"]`、`["mechanical"]`、`["electrical","mechanical"]`）。
  - 电气人员：`electrical`。
  - 机械人员：`mechanical`。
  - 超级管理员（需看全部）：`electrical,mechanical` 或两者都包含。
- **登录 / 当前用户接口**：返回 `material_scopes`（或前端用 `scopes`），前端据此控制菜单与路由。

这样实现**权限级隔离**：同一套前端，不同用户看到和可访问的仅是自身范围对应的菜单与页面。

### 2.2 前端隔离与“瘦身”手段

- **菜单**：仅渲染用户 `scopes` 中包含的物资类型对应菜单项（电气备件/电气领用、机械备件/机械领用等）。未授权类型不显示。
- **路由守卫**：  
  - 访问某路由时校验：若该路由属于某物资类型（如机械），而用户 `scopes` 中不包含该类型，则重定向到其有权限的默认页（如电气备件或电气领用）。  
  - 避免通过直接输入 URL 进入无权限的物资模块。
- **路由懒加载（关键）**：  
  - 电气相关页面（电气备件列表/领用/库存等）用 `() => import('@/views/...')` 打成单独 chunk。  
  - 机械相关页面同理，打成另一批 chunk。  
  - 用户只访问其有权限的页面，因此**只会加载对应 chunk**，不会加载无权限的物资类型代码，从而控制内存和 CPU，避免项目“虚胖”。

总结：**权限隔离 + 按范围显示菜单 + 路由懒加载**，在单项目、单部署的前提下，既做到电气/机械前端隔离，又保持运行时只加载当前用户需要的代码。

### 2.3 后端 API 校验（建议）

- 电气相关接口（如 `/spare-parts/*`）：校验当前用户 `material_scopes` 是否包含 `electrical`，否则 403。
- 机械相关接口（如 `/mechanical-spare-parts/*`）：校验是否包含 `mechanical`，否则 403。
- 通用接口（用户、操作日志、报表等）：可按需决定是否与物资类型绑定；若与类型无关，可不做 scope 校验。

这样即使有人绕过前端，直接调 API，也无法访问无权限的物资数据。

## 三、与“更好方法”的对比

| 方案 | 隔离方式 | 资源占用 | 复杂度 |
|------|----------|----------|--------|
| **A. 按用户 scope + 懒加载（推荐）** | 同一前端，菜单与路由按 scope 显示/守卫；电气/机械分 chunk，按需加载 | 单用户只加载其 scope 对应 chunk，占用小 | 中：加字段、守卫、懒加载 |
| B. 多前端部署（电气站/机械站） | 两套入口或两套构建，不同域名/路径 | 每个站点只含本类代码，但需维护两套或两套构建 | 高：部署与构建流程复杂 |
| C. 仅菜单隐藏，不懒加载 | 同一前端，只隐藏菜单，路由仍可被直接访问 | 所有物资类型 JS 都会加载，占用大 | 低但达不到“瘦身” |

推荐 A：在不拆成多项目、多部署的前提下，既能严格按权限隔离电气/机械，又能通过懒加载控制资源占用。

## 四、部署后新增物资类型（如生活消耗品）的两种思路

### 4.1 思路一：配置驱动（零/少量前端发版）

- **后端**：  
  - 增加“物资类型”配置（表或配置文件），例如：`code`、`name`、`form_schema`（表单字段定义）、`list_columns`（列表列定义）。  
  - 通用物资表：如 `materials(id, type, common_fields..., extra_json)`，或按类型分表但统一通过一层 API 抽象。  
  - 提供：`GET /material-types`、`GET /materials?type=xxx`、通用 CRUD（带 type）。
- **前端**：  
  - 一套**通用物资列表页 + 通用物资表单**，根据 `GET /material-types` 和对应 schema 动态渲染列表列和表单字段。  
  - 新增“生活消耗品”时：在后台增加类型配置与可选迁移，**前端无需发版**（或仅发小版本以支持新 type 的入口）。
- **优点**：部署后通过配置即可上新类型，扩展成本低。  
- **缺点**：列表/表单是通用 UI，表现力不如完全定制页面。

### 4.2 思路二：按类型分模块（小步发版）

- **后端**：每种物资类型独立表与 API（如现有电气、机械），新增“生活消耗品”则新增表与接口。
- **前端**：每种类型对应独立路由与组件（如电气、机械），新类型则新增路由 + 新页面（同样做懒加载）。  
- **用户**：在用户表 `material_scopes` 中增加新类型（如 `consumables`），菜单与路由守卫自动包含新类型；仅拥有该 scope 的用户会加载对应 chunk。
- **优点**：每种类型可完全定制 UI 与交互。  
- **缺点**：新增类型需要一次前端发版（但只是新增一个懒加载模块，不影响已有电气/机械 chunk）。

### 4.3 建议

- **短期**：采用**思路二**，与当前电气/机械实现一致，新增物资类型 = 新后端模块 + 新前端懒加载路由；通过 `material_scopes` 控制可见与可访问性，保持资源占用可控。  
- **中长期**：若希望“部署后仅通过配置就上新类型”，可再引入**思路一**的配置与通用 API/通用 UI，新类型优先走配置，特殊类型再保留独立模块。

## 六、数据与配置

- **用户表 `material_scopes`**：VARCHAR(200)，逗号分隔，如 `electrical`、`mechanical`、`electrical,mechanical`。空或 NULL 视为全部（电气+机械）。
- **仅电气人员**：`UPDATE users SET material_scopes = 'electrical' WHERE username = 'xxx';`
- **仅机械人员**：`UPDATE users SET material_scopes = 'mechanical' WHERE username = 'xxx';`
- **电气+机械（默认）**：`material_scopes = 'electrical,mechanical'` 或不设置。
- 修改后用户重新登录或前端重新请求 `/auth/me` 即可生效。
