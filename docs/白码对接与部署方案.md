# 备件管理系统 — 白码低代码平台对接与部署方案

本文档说明**在不修改本系统代码**的前提下，如何与白码低代码平台结合使用，以及可选部署方式。

---

## 一、结论摘要

| 问题 | 结论 |
|------|------|
| 能否把本项目「部署在白码里」运行？ | **不能**。白码是低代码开发平台，部署的是在白码内创建的应用，无法直接上传并运行本项目的 FastAPI + Vue 代码。 |
| 不改代码能否与白码一起用？ | **可以**。采用「备件系统独立部署 + 白码作为入口或数据扩展」的方式，无需改本系统代码。 |

---

## 二、本系统技术架构（便于理解对接方式）

- **后端**：FastAPI（Python），提供 REST API，端口 8888/8000
- **前端**：Vue 3 + Vite，构建后为静态资源，由 Nginx 或容器提供
- **数据**：MySQL（`spare_part_db`）、Redis、MinIO（图片）
- **部署**：Docker Compose 一键部署（见 [云服务器部署指南.md](云服务器部署指南.md)）

白码**不能**替代本系统的运行环境，只能通过**入口集成**或**数据对接**与现有系统配合使用。

---

## 三、不修改代码的两种方案

### 方案一：独立部署 + 白码入口集成（推荐）

**思路**：备件系统按现有方式独立部署（Docker/云服务器），白码仅作为**统一门户入口**，通过菜单链接或 iframe 打开本系统。

**步骤概览**：

1. **部署备件系统（不改代码）**  
   按 [云服务器部署指南.md](云服务器部署指南.md) 或现有 `docker-compose` 在自有服务器/云服务器上部署，获得访问地址，例如：  
   `https://spare.yourcompany.com`

2. **白码侧配置**  
   - 若白码支持**自定义页 / 外链 / 链接组件**：在菜单中新增一项「备件管理系统」，链接到 `https://spare.yourcompany.com`，用户从白码点击即可在新标签页打开本系统。  
   - 若白码支持**iframe 嵌入**：在自定义页中拖入 iframe 组件，地址填本系统首页地址（同上）。需注意：  
     - 本系统与白码若不同域，需保证备件系统已配置 CORS（本项目通过 `CORS_ORIGINS` 环境变量配置，将白码域名加入即可）。  
     - 若被嵌入，需允许被 iframe（部分浏览器/安全策略会限制，可后续按需在 Nginx 增加 `X-Frame-Options` 或 CSP，仅当白码确认为 iframe 方案时再考虑）。

3. **CORS 配置（仅当白码页面与备件系统跨域访问 API 时）**  
   若白码页面内通过前端直接请求备件系统 API（而非仅跳转打开本系统），则需在备件系统部署时设置环境变量，例如：  
   `CORS_ORIGINS=https://你的白码域名`  
   无需改代码，仅配置即可。

**优点**：零代码改动，本系统完整功能不变，白码仅做入口。  
**注意**：登录态独立。用户从白码点进备件系统后，需在备件系统再登录一次（除非后续单独做 SSO，会涉及额外配置或少量开发）。

---

### 方案二：独立部署 + 白码数据库对接（数据展示/扩展）

**思路**：备件系统照常独立部署并运行所有业务；白码通过**数据库对接**连接本系统使用的 MySQL，在白码内做报表、看板、统计或审批流等，不运行本系统代码。

**步骤概览**：

1. **备件系统独立部署**  
   同上，确保 MySQL 可从白码所在环境访问（同一内网或经安全组/白名单开放 3306，或通过白码支持的数据库代理方式）。  
   本系统使用的库名：`spare_part_db`（见 `docker-compose.yml` 及 `DEPLOY.md`）。

2. **白码平台 — 数据库对接**  
   - 在白码管理端：数据表 / 数据源 等位置添加「数据库对接」。  
   - 数据库类型选 **MySQL**。  
   - 连接信息与备件系统一致：主机、端口、库名 `spare_part_db`、账号密码（建议使用只读或专用账号，仅授予白码所需表的 SELECT 权限）。  
   - 选择需要展示或统计的表（如备件、领用记录等），按白码文档做「自动生成数据表」或映射。  
   - 白码侧避免使用表名 `user`、`role`（与白码系统表冲突，见白码官方说明）。

3. **使用方式**  
   - 在白码中基于对接表做：列表、图表、看板、导出、打印等。  
   - 增删改业务仍在本系统（FastAPI + Vue）完成；白码侧重查询与报表，或配合白码审批流做扩展。

**优点**：不改备件系统代码，数据统一来自本系统 MySQL，白码做数据分析与展示。  
**注意**：写操作若在白码进行，需自行评估与备件系统业务逻辑、约束的一致性；建议白码侧以只读或有限写为主。

## 四、白码 iframe 嵌入（HTTPS + 隐藏 IP）

将备件管理系统嵌入白码自定义页时，需注意两点：
1. **白码为 HTTPS**，iframe 也必须 HTTPS，否则浏览器会报 **Mixed Content** 错误并拦截
2. **不应暴露服务器 IP**，应使用与白码同域或子域名访问

下面提供三种方案（由优到劣），**优先采用方案 A（同域路径）**，与白码技术支持建议一致。

---

### 方案 A：同域路径代理（推荐，按白码技术支持建议）

**原理**：将备件系统挂到白码同一域名下的路径（如 `https://haodaapp.yxjyc.com/spare/`），由白码 Nginx 把 `/spare/` 反向代理到本机 8080。与白码同域、同 HTTPS，浏览器不报 Mixed Content，也不暴露 IP。

**效果**：
- iframe 地址：`https://haodaapp.yxjyc.com/spare/`
- 与白码同域，无跨域、无 Mixed Content
- 不暴露服务器 IP，无需单独证书或 DNS

#### 步骤

**1. 在白码 Nginx 中增加路径代理**

在**已提供 HTTPS 的** `haodaapp.yxjyc.com` 的 `server { }` 内，加入以下 `location`（可直接复制项目中的 `nginx/baima-path-proxy.conf.example` 内容）：

```nginx
location /spare/ {
    proxy_pass http://127.0.0.1:8080/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Forwarded-Host $host;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 86400;
    client_max_body_size 10M;
    proxy_buffering off;
    proxy_request_buffering off;
}
```

保存后执行：`nginx -t && nginx -s reload`

**2. 备件系统前端按「/spare/」base 构建并运行**

需让前端资源、API、MinIO 都带上前缀 `/spare/`，因此要用 `VITE_BASE_PATH=/spare/` 构建或启动。

- **若使用 Docker 部署**：在项目根目录 `.env` 或 `frontend/.env.production` 中增加一行：
  ```bash
  VITE_BASE_PATH=/spare/
  ```
  然后**重新构建**前端镜像并重启（见下方命令）。

- **若为开发模式**（frontend-dev 容器）：
  - 在 `docker-compose.yml` 的 `frontend-dev.environment` 中增加：`VITE_BASE_PATH: "/spare/"`
  - 重新构建并启动：`docker-compose up -d --build frontend-dev`，再重启 nginx：`docker-compose restart nginx`

- **本地构建**（在 frontend 目录）：
  ```bash
  cd frontend
  VITE_BASE_PATH=/spare/ npm run build
  ```
  再用当前项目的 Nginx 或静态服务托管 `dist`，保证对外访问路径为 `http://本机:8080/`（白码会代理 `/spare/` 到该地址）。

**3. 在白码 iframe 中填写地址**

在自定义页的 iframe 组件 **URL** 中填写：

```
https://haodaapp.yxjyc.com/spare/
```

```html
<iframe
  src="https://haodaapp.yxjyc.com/spare/"
  width="100%"
  height="100%"
  frameborder="0"
  allow="fullscreen"
  title="备件管理系统"
></iframe>
```

**4. 关于 favicon.ico 404**

若控制台出现 `favicon.ico 404`，多为浏览器对 iframe 页请求站根 favicon，可忽略，或确保 `frontend/public/favicon.ico` 存在（相对路径 `./favicon.ico` 会随 base 解析到 `/spare/favicon.ico`）。

> 配置模板见：`nginx/baima-path-proxy.conf.example`

---

### 方案 B：子域名反向代理

**原理**：为备件系统单独配置子域名（如 `spare.haodaapp.yxjyc.com`），由白码 Nginx 反向代理到本机 8080。

**效果**：iframe 地址为 `https://spare.haodaapp.yxjyc.com/`，不暴露 IP，但需 DNS 与证书（或通配符证书）。

**步骤**：配置 DNS、在白码 Nginx 中加入子域名 server 块（见 `nginx/baima-reverse-proxy.conf.example`）、重启 Nginx，iframe 填 `https://spare.haodaapp.yxjyc.com/`。**前端无需**设置 `VITE_BASE_PATH`（按根路径部署即可）。

---

### 方案 C：自签名证书 + IP 直连（应急）

**适用于**：无法修改白码 Nginx 且无法配置域名时的临时方案。

**缺点**：暴露服务器 IP；自签名证书需每个用户手动信任。

#### 步骤

**1. 生成 SSL 证书**

```bash
cd /opt/spare_part_system   # 按实际路径
bash ssl/generate-cert.sh
```

**2. 重启 nginx**

```bash
docker-compose restart nginx
```

**3. 信任自签名证书**

每个用户需在浏览器中先直接访问 `https://服务器IP:8443/` 并信任证书。

**4. 白码 iframe 地址**

```
https://172.24.69.125:8443/
```

---

### 方案 C：开发环境（仅本地）

若白码与备件系统在同一台机器且白码也是 HTTP：

```
http://localhost:5173
```

> 若白码为 HTTPS，本地开发也需要使用方案 A 或 B。

---

### 方案对比

| 维度 | 方案 A：子域名反向代理 | 方案 B：自签名 + IP |
|------|----------------------|---------------------|
| iframe 地址 | `https://spare.haodaapp.yxjyc.com/` | `https://IP:8443/` |
| 暴露服务器 IP | 否 | 是 |
| 用户需手动信任证书 | 否 | 是（每人一次） |
| 需修改白码 Nginx | 是 | 否 |
| 需 DNS 配置 | 是 | 否 |
| 需开放 8080/8443 端口 | 否（仅本机回环） | 是 |
| 推荐场景 | **生产环境** | 临时测试/应急 |

## 五、权限管理与白码对接

本系统与白码各有独立权限体系，若希望「以白码权限为准、免二次登录」或「统一管理用户」，可考虑以下对接方式。

### 5.1 本系统当前权限模型

- **认证**：JWT（`/auth/login` 登录后返回 token，后续请求携带 `Authorization: Bearer <token>`）
- **用户存储**：MySQL `users` 表（username、password_hash、role、status 等）
- **角色**：`admin`（超级管理员）、`electrical_admin` / `mechanical_admin`（电气/机械管理员）、`electrical_requisition_clerk` / `mechanical_requisition_clerk`（领用员）
- **物资范围**：admin 可访问电气+机械；电气角色仅电气；机械角色仅机械

### 5.2 白码平台权限概况

- 白码自带**用户管理**、**权限管理**、**菜单管理**
- 支持**外部用户**（游客，无需登录）
- 支持**第三方集成**（企业微信、钉钉等）
- 公开文档中未明确提供 OAuth/OIDC/SSO 等标准化协议接口，需向白码确认

### 5.3 对接思路（可选）

| 方案 | 思路 | 改动量 | 说明 |
|------|------|--------|------|
| **A. 维持双套登录（现状）** | 白码只做入口，用户进备件系统后再登录 | 无 | 实现简单，体验为二次登录 |
| **B. URL Token 透传** | 白码跳转/iframe 时在 URL 带 token 或 userId，备件前端用其调后端换 JWT | 小 | 需白码支持跳转时带参数；本系统新增「白码 Token 登录」接口 |
| **C. 白码 SSO / OAuth** | 白码作为 IdP，本系统作为 OAuth 客户端 | 中 | 需白码支持 OAuth2/OIDC；本系统增加 OAuth 登录流程 |
| **D. 用户数据同步** | 白码与备件系统共用或同步用户表，登录态通过共享 cookie/同域解决 | 大 | 需白码与备件同域或共享 session；架构改动较多 |

### 5.4 推荐实施路径

1. **短期**：采用**方案 A**，白码仅做入口，备件系统保持独立登录；将「备件管理」菜单权限控制在白码侧，仅授权用户可见入口。
2. **中期**：向白码确认是否支持：
   - 跳转/iframe 时携带**用户标识**或**临时 token**（用于方案 B）；
   - 或提供 **OAuth2/OIDC** 等 SSO 能力（用于方案 C）。
3. **若白码支持 URL 参数透传**：本系统可新增接口，例如  
   `POST /api/v1/auth/baima-login`，接收 `baima_user_id` 或 `baima_token`，在本系统 users 表中映射后签发 JWT，实现免登录。

### 5.5 需向白码确认的事项

- 白码是否支持**自定义跳转 URL 带参数**（如 `?user_id=xxx` 或 `?ticket=xxx`）？
- 白码是否提供 **SSO / OAuth / OIDC** 或类似能力供外部系统集成？
- 白码能否在 iframe 嵌入场景下，通过 **postMessage** 向子页面传递当前登录用户信息？

确认以上能力后，可据此选择方案 B/C 并设计具体对接实现。白码技术支持：020-88520693，business@bnocode.com。

## 六、方案对比与选择

| 维度         | 方案一：白码入口集成     | 方案二：白码数据库对接       |
|--------------|--------------------------|------------------------------|
| 代码修改     | 无                       | 无                           |
| 备件系统部署 | 独立部署（现有方式）     | 独立部署（现有方式）         |
| 白码作用     | 门户入口（链接/iframe）  | 数据展示、报表、看板、扩展   |
| 登录         | 进备件系统后独立登录     | 白码与备件系统各自登录       |
| 数据一致性   | 无额外考虑               | 同库，注意写权限与业务约束   |
| 适用场景     | 统一从白码进入各系统     | 需要在白码里看备件数据/报表  |

- 仅需**从白码跳转到备件系统**：选方案一。  
- 需要在**白码内看备件数据、做报表或扩展**：选方案二，或方案一 + 方案二同时使用。

## 七、部署清单（不修改代码）

1. **备件系统**  
   - 按 [云服务器部署指南.md](云服务器部署指南.md) 或现有 Docker Compose 部署。  
   - 记录对外访问地址（如 `https://spare.yourcompany.com`）。  
   - 若采用方案一且白码与备件系统跨域调用 API：在 `.env` 中设置 `CORS_ORIGINS=白码域名`。

2. **白码侧**  
   - **方案一**：在菜单或自定义页中配置「备件管理系统」链接或 iframe 地址为上述备件系统地址。  
   - **方案二**：在白码中添加 MySQL 数据源，连接 `spare_part_db`，配置对接表并制作页面/报表。

3. **安全**  
   - 数据库对接使用专用账号、最小权限（如只读）。  
   - 生产环境务必 HTTPS、强密码与 `SECRET_KEY`，见 [DEPLOY.md](../DEPLOY.md)。
